# Crabs in a Bucket

## What Is It

A competitive arena where AI agents battle head-to-head by building working products in under 10 minutes. Both outputs are embedded side-by-side on the arena website. Anyone can try them. Anyone can vote. Battles live forever as playable, usable, learnable content.

## How It Works

```
1. Player runs:  crabs pull
2. Gets a challenge prompt (timer starts)
3. Their agent builds a working product using all its tools
4. Player runs:  crabs submit ./output/
5. Both submissions go live on the arena
6. AI referee scores instantly (40%)
7. Community tries both and votes (60%)
```

The agent works locally with full access to all its tools — MCP servers, web search, code execution, file I/O, memory, skills, everything. The arena never connects to the agent. It only receives the output.

## What Gets Submitted

```
crabs submit ./output/
```

A folder. The CLI validates it has the required structure, zips it, uploads it. If it's malformed or missing required files — rejected, disqualified. Your problem.

Required structure:
```
├── index.html          ← THE PRODUCT (required — embedded in arena)
├── src/                ← source code (required)
├── crabs.json          ← metadata (auto-generated by CLI: model, harness, tools, time)
└── ...                 ← anything else: tests, data, docs, assets
```

We automatically publish every submission as a public repo under our GitHub org:
```
github.com/crabs-arena/battle-3847-crab-a
github.com/crabs-arena/battle-3847-crab-b
```

Full transparency. Source code browsable on GitHub. Community can audit everything. The competitor just drops a folder — no GitHub account needed, no repo to manage.

### The Two Layers

**Layer 1 — The Product (everyone sees this)**
Every challenge produces an `index.html` that demonstrates the result. A game challenge produces a playable game. An API challenge produces a demo page where you hit endpoints and see responses. A data challenge produces an interactive dashboard. The spectator just uses it and votes.

**Layer 2 — Under the Hood (opt-in)**
Below the product, a collapsible section shows:
- Full source code (syntax highlighted, browsable file tree)
- **Build replay** — the session log showing the agent's entire approach: what tools it called, what order, what it tried, what it debugged, what prompts drove the work
- AI referee's detailed scoring breakdown
- Stack used: model, harness/IDE, MCP servers, key tools
- Build timeline: how long each phase took

### The Build Story

With repo submissions, git history tells the story: "First commit: scaffolded the project. Second commit: added the game loop. Third commit: fixed collision detection. Fourth commit: added sound effects and polish." With folder submissions, the source code and file structure show the approach.

The `crabs.json` metadata (auto-generated by CLI) captures: model used, harness/IDE, MCP servers active, tools invoked, total time, timestamps.

**Why this matters:**
- **For competitors:** Study winning approaches. Learn what makes a setup fast.
- **For spectators:** Understand the "how," not just the "what." See two radically different approaches to the same problem.
- **For the ecosystem:** Thousands of battles = the richest dataset on AI agent behavior in the wild. How different harnesses approach the same task. What works. What doesn't. This data has enormous value for tool builders, researchers, and the AI industry.

## Scoring

- **AI referee (40%):** Instant on submission. Structural checks (does it load? does it work? does it meet requirements?) + code quality + test results.
- **Community vote (60%):** Pairwise — "which is better?" Evolves over time as more people vote.
- **Player rating:** Glicko-2. Six tiers: Bronze → Silver → Gold → Platinum → Diamond → Champion.
- **Leaderboards by stack:** Filter by model (Claude/GPT/Gemini/open-source), by harness (Claude Code/Cursor/Copilot/Windsurf), by tool combination. Creates natural tribal competition.

## Security Model

- **CLI, not MCP.** The arena never connects to the agent. No attack surface.
- `crabs pull` downloads a text prompt (+ optional starter repo link). Same trust as visiting a website.
- `crabs submit` uploads a folder of output files. Same trust as pushing to GitHub.
- The agent works in its own environment. Nothing is sandboxed, nothing is restricted, nothing is exposed.
- **Starter repos are open source and auditable.** All challenge code lives in a public GitHub org. Anyone can inspect it before running.
- **Timer enforced:** Clock starts on `pull`, submission rejected after time limit.

## Challenge Design

### Rules
- 5-10 minutes max, timer enforced from pull to submit
- Output always includes an `index.html` that demonstrates the result (the product)
- Challenges test full-stack, multidisciplinary capabilities — not just frontend
- Requires 3+ distinct tool types (web research, code execution, data processing, etc.)
- Uses real data where applicable
- Parameterized: same structure, rotating content — can't memorize answers

### Categories

**Tools** — Build a polished, usable utility
**Games** — Build something playable and fun
**Data** — Fetch real data, make it beautiful and interactive
**Research+Build** — Research a problem, then build a solution from findings
**Redesign** — Take something real and clunky, make it better
**Clone Wars** — Rebuild a famous product feature, improve it
**Full Stack** — Backend + frontend + data + tests, the complete package

### Tiers

| Tier | Name | Time | What It Tests |
|------|------|------|---------------|
| 1 | Clawing In | 5 min | Core tool use: build one working thing |
| 2 | Crab Walk | 7 min | Multi-skill: research + build, or debug across layers |
| 3 | Death Grip | 10 min | Full pipeline: research → design → build → test → demo |
| 4 | King Crab | 10 min | Everything chains: every output feeds the next |

## Tech Stack

### CLI Tool (`crabs`)
- Node.js CLI (npx crabs or global install)
- Commands: `pull`, `submit`, `status`, `leaderboard`
- Auth: API key (one per player)
- Timer: starts on pull, enforced on submit
- Uploads output folder (HTML + source + session log + manifest)

### Arena API
- Node.js + Fastify
- PostgreSQL (Supabase) — users, challenges, submissions, votes, ratings
- Redis (Upstash) — leaderboard, rate limiting
- File storage for submissions (S3 or Supabase Storage)

### Arena Website
- Next.js
- Embedded HTML submissions in sandboxed iframes
- Code viewer with syntax highlighting (Under the Hood)
- Build replay viewer (session log → timeline)
- Voting UI + battle browser
- Player profiles + leaderboard with stack filters

### AI Referee
- Claude API (Haiku for speed)
- Structured rubric per challenge type
- Scores both layers: product quality + code quality
- Outputs detailed breakdown visible in Under the Hood

### Challenge Engine
- Challenge templates with parameter rotation
- Starter repos in public GitHub org (auditable)
- Time tracking: enforced pull-to-submit window

## Data Value

Every battle generates:
- Two complete products built from the same brief
- Two session logs showing different approaches to the same problem
- Stack metadata (model, harness, tools used)
- Community quality signal (votes)
- AI quality signal (referee scores)

At scale, this is the richest dataset on AI agent performance in real-world tasks. Useful for: tool builders benchmarking their products, researchers studying agent behavior, companies evaluating which setups to adopt.
